## 项目定位

异次元店铺系统是一个原生 PHP 的虚拟发卡/个人店铺系统，文档中称其为荔枝店铺系统 3.0 的重构版本。它不是前后端分离项目，也不是 npm 构建型前端项目。

核心技术和能力：

- PHP 原生开发，要求 PHP 8 及以上。
- 数据库层使用 Eloquent ORM。
- 模板层使用 Smarty 3.1，同时也有 PHP 原生渲染。
- 会话使用 PHP session。
- 支持支付插件、商品销售、会员/商户等级、分站、推广返佣、共享店铺、应用商店、模板和插件扩展。
- 可通过插件扩展发货逻辑，例如对接第三方网站、自动授权、余额充值、论坛/社区权益开通等。

官方文档有明确法律提示：项目基于 MIT 协议开源、免费，初衷是学习研究；实际使用必须遵守法律法规，并注意商业资质和合规风险。

## 官方推荐部署方式

官方文档推荐的是常规 PHP 站点部署，而不是 Docker。

最低环境：

| 组件 | 要求 |
| --- | --- |
| PHP | `>= 8.0` |
| MySQL | `>= 5.7` |
| Web Server | Apache、Nginx、IIS 均可 |

源码获取方式：

- 从 GitHub 下载源码到服务器。
- 从官方文档提供的最新版 zip 下载入口获取压缩包。
- 可执行命令的环境可使用 Composer 创建项目：

```bash
composer create-project lizhipay/acg-faka
```

Web Server 配置：

- Apache：项目根目录自带 `.htaccess`，官方说无需额外配置。
- Nginx：需要配置伪静态，把不存在的路径转给 `index.php?s=...`。
- IIS：需要 URL Rewrite 规则。

Nginx 伪静态核心规则：

```nginx
location / {
    if (!-e $request_filename) {
        rewrite ^(.*)$ /index.php?s=$1 last;
        break;
    }
}
```

部署完成后访问首页进入安装向导。安装完成后的后台地址：

```text
https://你的域名/admin
```

## Docker 部署判断

官方没有把 Docker 作为主推部署方式，但 Docker 可以复刻官方环境，因此可行。

推荐容器结构：

- `nginx`：静态资源、入口转发、伪静态。
- `php-fpm`：PHP 8 运行环境。
- `mysql`：数据库，版本建议 5.7 或 8.0。
- 可选 `redis`：session 加速。

需要持久化的数据：

- MySQL 数据目录。
- `config/database.php`
- `config/store.php`，如果启用应用商店。
- `kernel/Install/Lock`
- `runtime/`
- `assets/cache/`

二次开发场景下，建议把官方“后台云更新”和自己的 Git 部署流程分开考虑。后台云更新可能覆盖二开代码，生产发布建议走自己的 Git 仓库和自动化部署。

## 版本管理

官方使用 `x.y.z` 版本规则：

- `x`：重大版本，可能包含核心重构或大量破坏性 API 变化。
- `y`：主要功能迭代版本，可能涉及公开 API 的不兼容变化。
- `z`：兼容修复版本，主要用于 bug、安全问题或不影响既有用法的新功能。

版本支持计划：

官方推荐升级方式：

- 优先在网站后台左下角点击版本号，检测并自动升级。
- 手动升级需要按版本顺序下载升级包，把 `file` 目录覆盖到站点根目录。
- 如果升级包包含 `update.sql`，需要把 SQL 中的 `__PREFIX__` 替换为实际表前缀，再执行数据库升级。
- 升级完成后修改 `/config/app.php` 中的 `version`。

二开注意：

- 官方后台升级更适合未二开的标准站点。
- 二开站点建议先在测试环境合并升级包，确认模板、静态资源、数据库结构和插件兼容后再上线。

## 已知升级问题

### 0.8.4-beta / 0.8.5-beta

现象：

- 更新后前台点击商品会出现黑色提示条。

原因：

- 更新过程中的 `update.php` 可能被 OPcache 缓存，导致升级逻辑没有真正生效。

修复思路：

- 官方提供补丁包，解压到网站根目录后访问 `patch.php` 修复，修复后删除补丁文件。
- 也可以手动执行 SQL，为商品表新增 `maximum` 字段。
- 如果安装时修改了表前缀，需要把 SQL 里的默认前缀替换为实际前缀。

涉及字段：

```sql
maximum int UNSIGNED NULL DEFAULT 0
```

### 0.8.8-beta

现象：

- 更新后部分浏览器无法登录后台。

修复思路：

- 使用 Chrome 登录后台，再升级到 `0.8.9-beta`。

## 会员模块

会员等级关注两个核心字段：

- 累计充值：用户充值累计达到门槛后自动升级。
- 会员折扣：会员价按折扣计算，例如原价 100、折扣 10%，实际价格为 90。

会员等级不只是普通会员权益，也可作为代理等级的基础。别人通过 API 或店铺共享对接你的商品时，也会受会员等级价格体系影响。

商户等级用于供货商/店铺能力控制。一个用户可以作为会员，也可以被赋予商户能力。

商户等级常见控制项：

- 开通商户功能需要支付的价格。
- 商品成交后的抽成比例。
- 分站销售主站商品时的佣金。
- 是否允许开通分站。
- 是否允许供货，即创建分类和商品。
- 是否允许绑定独立域名。

## 分站模块

分站用于让商家通过自己的域名或子域名经营店铺。

显示主站商品：

- 商家可在用户中心的“我的店铺”中设置是否显示主站商品。
- 不显示主站商品时，分站主要销售商家自己上架的商品。

子域名模式：

- 后台“网站设置 -> 其他设置”需要配置主站域名和 DNS-CNAME。
- 假设主站域名为 `abc.com`，商家子站可形如 `xxx.abc.com`。
- 域名服务商处需要配置泛解析，例如 `*.abc.com` 指向服务器 IP。
- 服务器或虚拟主机还需要绑定这个泛域名，例如 `*.abc.com`。

独立域名模式：

- 商家拥有独立域名且等级允许绑定时，可把自己的域名 CNAME 到站点指定地址。
- 如果解析后仍无法访问，通常要检查默认站点是否指向当前店铺程序。

## 供货系统

供货系统有两种主要玩法：

- 商家供货到主站：给商户等级开启供货权限，商家添加分类和商品后，主站可在后台设置显示某个商家的商品。
- 商家自营店铺：商家绑定子域名或独立域名，并在自己的店铺隐藏主站商品后，就成为较独立的自营店。

如果用户是在分站注册的账号，会归属到对应分站，后续消费会触发推广返佣逻辑。

## 推广返佣

### 3.1 及以下

旧版本采用固定比例模式，后台有 1 级、2 级、3 级返佣比例。文档通过多级推荐链演示了返佣层层分配。

要点：

- 直接推荐人先获得一部分返佣。
- 更上级可从下级返佣中继续按比例获得一部分。
- 0.8.1-beta 增加分站加价后，分站加价会影响 1 级返佣计算。

### 3.2 及以上

新版采用会员差价模式。

核心逻辑：

- 不再主要依赖固定比例。
- 通过不同会员等级的商品专属价格形成差价。
- 下级按更高价格购买，上级按更低拿货价计算，差额成为上级收益。
- 多级场景下，系统逐层比较上下级价格差，只要存在差价就继续向上结算。

示例理解：

- 普通用户支付 100。
- 直接上级的拿货价为 80，则直接上级得到 20。
- 更上级拿货价为 70，则更上级得到 10。

二开结论：

- 新版本代理体系应优先围绕会员等级价、商品独立会员价、SKU/种类加价设计。
- 不建议在不了解价格差逻辑的情况下硬改返佣比例。

## 店铺共享/店铺对接

店铺共享用于对接其他站点商品。

流程：

1. 到目标店铺注册账号。
2. 在用户主页获取商户 ID 和商户密钥。
3. 回到自己店铺后台，进入“店铺共享”，新增远程店铺。
4. 填写对方店铺地址、商户 ID、密钥。
5. 使用导入功能把远程商品导入本地分类。
6. 导入后的商品默认下架，可修改价格后再上架。

客户购买链路：

- 客户在你的店铺下单。
- 你的系统使用店铺共享接口到远程站点余额购买。
- 远程站点返回卡密或插件发货结果。
- 你的系统把发货结果保存到本地订单并展示给客户。

关键要求：

- 对接方账号必须有足够余额，否则无法完成远程余额下单。
- 店铺共享支持多层对接，最终发货仍来自最上游实际供货站点。
- 代理等级可通过会员等级价格体系实现。
- 文档提到支持远程商品定价同步。

## 商品管理

商品管理重点能力：

- 会员等级定价：每个商品可为不同会员等级设置独立价格和批发价格，适合代理体系。
- 自动发货：购买后自动发卡密。
- 手动发货/插件发货：可固定展示发货信息，也可通过插件对接外部系统后返回发货内容。
- 发货留言：用于售后群、下载地址、说明等。
- 邮件发送：使用前必须先在网站邮箱设置中配置并测试成功。
- 查询密码：开启后游客查单/购买需要密码；已登录用户可免输入部分信息。
- 优惠券：支持抵扣金额。
- 限时秒杀：限定时间内可购买，前台显示倒计时。
- 卡密预选：用户可选择指定卡密，常用于游戏账号交易；导入卡密时可用正则处理预选信息。
- 强制登录：购买前必须登录。
- 限购数量：购买达到限制后不能继续买，适合活动控制。
- API 对接：开启后其他站可拉取并对接该商品。
- 隐藏库存：前台不显示准确库存，改为范围描述。
- 控件：为商品添加输入框、下拉框、多选框、单选框、文本域等，让用户提交额外信息。

控件扩展数据：

- 下拉框、多选框、单选框支持扩展数据。
- 扩展数据采用键值对形式，并用逗号分隔多项。
- 例如可配置为多个显示项与内部值的映射。

## 商品配置参数

商品配置参数位于添加/修改商品页面，总后台和供货商后台均可使用。它是商品扩展能力的关键。

### 批发参数

节点名：

```ini
[wholesale]
```

用途：

- 按购买数量设置单价。
- 达到指定数量后使用对应批发单价。

示例结构：

```ini
[wholesale]
5=90
10=80
20=65
```

含义：

- 购买 5 个及以上，单价 90。
- 购买 10 个及以上，单价 80。
- 购买 20 个及以上，单价 65。

### 商品种类参数

节点名：

```ini
[category]
```

用途：

- 同一商品提供多个种类和对应价格。
- 适合月卡、季卡、年卡等类型。

示例结构：

```ini
[category]
月费=20
季费=70
年费=150
```

### 种类成本参数

3.4.3 起新增，用于利润计算。

节点名：

```ini
[category_cost]
```

示例结构：

```ini
[category_cost]
月费=10
季费=30
年费=80
```

### 多 SKU 参数

3.1.1 起支持。

节点名：

```ini
[sku]
```

用途：

- 给商品配置多个规格维度。
- 每个规格项可对基础价格加价。
- 如果商品同时配置了种类价格，则 SKU 加价叠加到所选种类价格上。

示例结构：

```ini
[sku]
机身颜色.黑色=200
存储容量.256GB=400
套餐类型.官方标配=0
```

### SKU 成本参数

3.4.3 起新增，用于利润计算。

节点名：

```ini
[sku_cost]
```

示例结构：

```ini
[sku_cost]
机身颜色.黑色=100
存储容量.256GB=200
套餐类型.官方标配=0
```

### 种类批发参数

节点名：

```ini
[category_wholesale]
```

用途：

- 对具体商品种类单独设置批发价。
- 格式为 `种类.数量=单价`。

示例结构：

```ini
[category_wholesale]
天卡.5=9.88
天卡.10=8.88
周卡.3=60.88
```

高级用法：

- 商品配置参数可与会员独立定价、会员独立配置结合，控制不同等级看到的种类、价格和批发规则。

## 其他设置

官方重点说明了“自定义支付回调域名”。

适用场景：

- 主站域名接入 CDN。
- CDN 或防火墙导致支付回调慢、失败或不稳定。

解决思路：

- 额外绑定一个直连服务器的 IP + 端口或未走 CDN 的域名。
- 在“网站设置 -> 其他设置”中配置自定义支付回调域名。

安全判断：

- 某些表单提交型支付可能暴露回调域名，风险主要是泄露真实服务器 IP 或非 CDN 域名。
- 内部请求式支付插件通常不会暴露。
- 官方总体倾向推荐配置，以提高回调稳定性。

## API 对接

API 用于外部程序拉取商品、读取库存、询价、下单和查询订单。需要先开启共享对接。

基础规则：

| 项 | 内容 |
| --- | --- |
| 请求方式 | `POST` |
| 推荐格式 | `application/x-www-form-urlencoded` |
| 鉴权 | `app_id` + `sign` |
| 成功状态码 | `code = 200` |
| 失败状态码 | 通常 `code = 0`，错误见 `msg` |

签名规则：

1. 移除请求参数中的 `sign`。
2. 按参数名升序排序。
3. 值为空字符串的参数不参与签名。
4. 用表单查询串拼接参数。
5. 在末尾追加 `&key=商户密钥`。
6. 对最终字符串做 `urldecode` 后计算 MD5。

公共请求参数：

| 参数 | 必选 | 说明 |
| --- | --- | --- |
| `app_id` | 是 | 商户 ID，在用户中心商户资料中获取 |
| `sign` | 是 | 按签名规则生成的签名 |

公共响应字段：

| 字段 | 说明 |
| --- | --- |

### 推荐对接流程

1. 用 `items` 或 `item` 获取商品。
2. 如果有种类或 SKU，先确定 `race` 和 `sku`。
3. 如果支持预选，调用 `draftCard`，需要时再调用 `draft`。
4. 下单前调用 `stock`、`valuation` 或 `inventoryState` 校验库存和价格。
5. 调用 `trade` 下单。
6. 需要轮询发货时，用 `query` 查询订单。

## 卸载应用商店

官方说明项目根目录 `kernel/Plugin.php` 是加密闭源文件，用于验证付费插件。

处理方式：

- 如果介意闭源文件，可以删除 `kernel/Plugin.php`。
- 删除后系统主体仍可运行。
- 删除后应用商店和付费插件验证能力不可用，付费插件也无法正常生效。

二开建议：

- 如果不计划使用官方应用商店，离线化部署可以考虑删除该文件。
- 如果后续想用官方插件或模板，不要删除。

## Redis 性能优化

官方指出站点卡顿的一个常见原因是 PHP session 文件锁。

问题背景：

- Web Server 会并发处理多个请求。
- PHP 默认文件 session 会在 `session_start()` 后加锁。
- 同一用户多个并发请求可能互相等待，导致前后台体验卡顿。

优化方案：

- 安装 Redis 服务。
- 为 PHP 8 安装 Redis 扩展。
- 修改 PHP session 配置，让 session 存储到 Redis。

无密码 Redis 配置示例：

```ini
session.save_handler = redis
session.save_path = "tcp://127.0.0.1:6379"
```

有密码 Redis 配置示例：

```ini
session.save_path = "tcp://127.0.0.1:6379?auth=你的redis密码"
```

执行后需要重启 PHP。

注意：

- 这是性能优化，不是安装必需项。
- 虚拟主机通常无法自行安装 Redis，因此官方不保证虚拟主机能使用这项优化。
- 如果迁移旧站且旧站使用 Redis session，新环境也要配置对应扩展和服务。

## 疑难排查

官方建议遇到问题时先收集证据。

排查要点：

- 查看网站根目录或相关目录下的日志，例如 `runtime.log`。
- 回忆问题出现前做过什么：改配置、升级环境、点击某按钮、安装插件等。
- 准备截图和关键报错，再向社区提问。

常见迁移问题：

- 网站搬家后异常，优先检查 `config/database.php`。
- 如果旧站使用过 Redis session，新站必须安装 Redis 扩展并启动 Redis 服务。

## 对本项目二开的直接影响

### 本地调试

本项目适合按 PHP 传统站点方式调试：

- 不需要 npm 构建。
- 前台模板主要在 `app/View/User/Theme/Cartoon`。
- 前台 CSS/JS 主要在 `assets/user`。
- 后台模板主要在 `app/View/Admin`。
- 后台 CSS/JS 主要在 `assets/admin`。

### 部署

官方环境要求可以映射为：

- Nginx + PHP-FPM + MySQL。
- 可选 Redis。
- PHP 需要 `pdo_mysql`、`gd`、`curl`、`zip`、`openssl`、`session`、`json` 等能力。

### 自动更新

官方自动升级适合标准安装；二开项目更适合：

- 自己的 GitHub 仓库管理代码。
- 本地测试后推送。
- 服务器通过脚本或 CI/CD 拉取更新。
- 线上保留 `config/database.php`、`runtime/`、`assets/cache/` 等运行数据。

### 风险点

- 官方后台升级可能覆盖二开文件。
- 付费插件依赖 `kernel/Plugin.php`。
- 共享店铺对接依赖余额、商户 ID、商户密钥和 API 签名。
- 分站依赖 DNS 泛解析、服务器域名绑定、默认站点配置。
- Redis 优化迁移时必须同步环境配置。
